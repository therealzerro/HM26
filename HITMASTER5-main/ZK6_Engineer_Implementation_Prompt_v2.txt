0) ORIENTATION — WHAT THIS APP DOES
HitMaster is a decision-support app that turns lottery history into explainable “heat” and ranked K‑slates. It ingests four kinds of inputs:
• BOX & PAIR DATASETS = history imports (training baselines)
• DAILY INPUT DATASETS = rescoring deltas (only the combos that actually hit today)
• LEDGER DATA = authoritative straight results stream (Midday/Evening/All‑Day), used for recency logic and pair derivations

Minimum viable data: 11 classes each with a 1‑year horizon (H01Y): 1×Box class + 10×Pair classes. The slate MUST run on H01Y‑only and automatically improve as H02Y…H10Y arrive per class.


1) DATA ROLES & EXAMPLES — READ CAREFULLY
A) BOX (history) — unordered digit‑sets {A,B,C}
Columns: Combo (or ComboSet), TimesDrawn, Expected, LastSeen (ET date), DrawsSince (non‑negative int)
Example rows:
  000  28  26.49  Aug 3, 2025   3545
  001  65  79.47  Sep 11, 2025   705
  002  68  79.47  Sep 18, 2025   185

B) PAIRS (history; 10 pair classes share this shape)
Columns: Pair (00–99), TimesDrawn, LastSeen, DrawsSince
Example rows:
  00  678  Sep 20, 2025  19
  01  1348 Sep 20, 2025  18
  02  1443 Sep 20, 2025  59

C) DAILY INPUT (rescoring delta; same shape as BOX; only sets that hit today)
Columns: Combo, TimesDrawn, Expected, LastSeen, DrawsSince
Example rows:
  006  1  0.22  Sep 20, 2025  61
  008  1  0.22  Sep 20, 2025  19

D) LEDGER (results; authoritative)
Columns: Game, DrawDate, Results (straight digits), Session (Midday/Evening) by jurisdiction
Example rows:
  Arizona  Pick 3      Sat, Sep 20, 2025   2-8-4
  Arkansas Cash 3 Mid  Sat, Sep 20, 2025   3-2-2
  Arkansas Cash 3 Eve  Sat, Sep 20, 2025   4-3-0
  California Daily 3 Mid Sat, Sep 20, 2025 3-0-2
(Use uploaded “Drawing Times” as guidance for session scheduling.)

E) TEN PAIR TYPES (classes)
Use the uploaded “Pair Classes” guide to define classes:
  1) Front Pair Straight (AB)
  2) Back Pair Straight (BC)
  3) Split Pair Straight (AC)
  4) Front Pair Box (unordered {A,B})
  5) Back Pair Box (unordered {B,C})
  6) Split Pair Box (unordered {A,C})
  7) Front Pair from Box Combination (AB after box‑sort)
  8) Back Pair from Box Combination (BC after box‑sort)
  9) Split Pair from Box Combination (AC after box‑sort)
 10) Any Position Box (all unordered pairs {A,B},{A,C},{B,C} in each draw)

4) INGESTION WORKFLOWS (ADMIN‑ONLY; IDEMPOTENT)
A) HISTORY IMPORTS (Box + Pair classes; H01Y…H10Y, one year per file)
   1. Validate required columns & types; DrawsSince ≥ 0; LastSeen parseable (YYYY‑MM‑DD).
   2. Normalize keys: zero‑pad pairs “7”→“07”; sort box digits; strip commas; uppercase jurisdictions.
   3. Deduplicate by (class_id,horizon,scope,key,source); newest file wins.
   4. Persist rows into datasets_box / datasets_pair.
   5. Compute & store per‑class per‑horizon percentiles for ds_raw; winsorize at P99.

B) DAILY INPUT (rescoring deltas)
   1. Same schema as BOX; rows are today’s hits only.
   2. For each Combo ABC: reset DS=0 for its box set in active horizons; increment DS+1 for all other sets; emit pair events (see PAIR INTERPOLATION).

C) LEDGER (authoritative results)
   1. Upsert each jurisdiction’s {date_et, session, result_digits}; derive ComboSet (sorted).
   2. Advance the draw clock: DS+1 globally for non‑hit keys inside the active window.
   3. Emit pair events per result (Front AB, Back BC, Split AC, and AnyPos unordered pairs).

Watermarks/idempotency: hash (jurisdiction|date_et|session|result_digits) and ignore duplicates.


5) PAIR INTERPOLATION (KEEP PAIRS “LIVE” DAILY)
Every time a result lands (from DAILY INPUT or LEDGER):
• Fire events:
  – Front(AB), Back(BC), Split(AC)
  – AnyPos unordered pairs {A,B},{A,C},{B,C}
• For pairs that fired: set DS=0; for all others in that class: DS+=1 within active horizons.
• Maintain per‑class winsorized DS percentiles; update incrementally after each batch (nightly full refresh OK).
Outcome: Pair classes reflect today’s movement between yearly imports.


6) NORMALIZATION & HORIZON BLENDING
Per class × scope × horizon (H01Y…H10Y):
• Winsorize ds_raw at P99 to cap outliers.
• Normalize to [0,1] using percentile ranks (preferred). Store maps for reuse.
• If multiple horizons exist for a class, blend with short‑heavier geometric weights; weights sum to 1.
• If only H01Y exists, use that normalized value directly.
• Missing classes/horizons contribute 0 and are labeled “partial data” in UI.


7) ZK6 SCORING (BOX‑FIRST, PAIR‑ASSISTED)
Compute for each straight combo ABC (after normalization and blending):
• BOX*  = percentile DS for ComboSet {A,B,C}
• PBURST* = best‑order average of Front*(AB), Back*(BC), Split*(AC)
• CO*   = AnyPos unordered pair pressure average over {A,B},{A,C},{B,C}
Suggested weights to start (tuneable in Admin “Scoring Lab”):
• BOX 0.40, PBURST 0.40, CO 0.20   (alternate conservative: 0.85/0.10/0.05)
Indicator(ABC) = 0.40*BOX* + 0.40*PBURST* + 0.20*CO*

Rails (composition & safety):
• Same‑day exclusion: remove any straight or box that hit today (from Ledger).
• Quotas for K6: Singles=4, Doubles=2 (Triples off by default or gated).
• Pair repetition cap: max 2 items sharing the same TopPair in K6.
• Recency rules: exclude last‑24h global hits; apply Mid→Eve / Eve→Mid same‑day safeguards.


8) SLATE ASSEMBLY & TEMPERATURE
• Compute Indicator for all eligible combos; sort desc; fill K6 with quotas and pair caps; produce a 10‑item backfill.
• Convert Indicator to Temp 0–100 using distribution cut‑points (e.g., p10→0, p90→100, clipped).
• UI micro‑bars on each card: BOX / PBURST / CO (0–100). Show “BestOrder” and “TopPair” chips.
• Winners in Recent get a ✅ tag and are excluded from new slates that day.


9) REGENERATION TRIGGERS
Regenerate K‑slates when:
• An Admin imports a new horizon (any class, any year) → recompute that class’s blend and publish.
• An Admin ingests a new Ledger batch for a scope (Midday/Evening) → apply DS updates, exclusions, publish.
• Admin taps “Re‑Generate Slate”. Each publish stores a slate_snapshot with ET timestamp and horizons_present map.
